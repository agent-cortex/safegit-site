---
title: Approval Flow
description: How SafeGit turns a commit into a Safe-backed approval.
---

SafeGit has four moving pieces: CLI, API, Postgres, and CI.

## Flow

1. Developer runs `safegit add ...` and `safegit commit -m ...`; SafeGit delegates those commands to `git`.
2. Developer runs `safegit sign`.
3. The CLI reads Git metadata and repo Safe configuration for `HEAD`.
4. SafeGit creates a typed `GitCommitApproval` payload.
5. The payload and status are stored in Postgres.
6. Safe owners open `/approve/:id` and sign typed data.
7. The API verifies signatures and counts unique valid owners.
8. `safegit push` checks approval, then delegates to `git push`.
9. CI can also run `safegit verify --ref HEAD`, and GitHub branch protection can require that check.

## Payload shape

SafeGit approvals include replay protection fields, not just a commit hash:

- repo host, owner, and name
- branch and ref
- commit SHA and tree SHA
- author and committer
- Safe address and chain ID
- nonce and expiry

## Why EIP-712

Owners should see what they are approving. A structured payload is reviewable. A naked hash is a footgun wearing a trench coat.

## Shared state

Local-only state is fine for demos. Teams need shared state so signers, CI, and GitHub checks all see the same approval.

SafeGit uses Postgres for that shared state.
